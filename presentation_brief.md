# Presentation Brief

## Communication Job

By the end of the deck, the audience should believe:

- this candidate can turn a legal/privacy requirement into a product system
- the thinking is grounded in user behavior, business reality, and friction
- the proposal is specific enough to ship, measure, and defend

This is not a marketing deck and not a privacy manifesto.

It is a product judgment deck.

## Narrative Shape

The story should move in this order:

1. WhatsApp has a real trust problem to solve as the product expands.
2. The issue is not "users need more privacy settings."
3. The real issue is that different data uses need different product treatment.
4. A better system separates essential processing, continuity features, and optional business/discovery uses.
5. That system can improve clarity and control without breaking the core messaging experience.
6. Success depends on consent quality, not just consent quantity.

Every slide should create the need for the next slide.

## Tone Rules

The deck should sound:

- sharp
- calm
- specific
- slightly opinionated
- human

The deck should not sound:

- breathless
- slogan-heavy
- legalistic
- startup-pitchy
- management-consulting-generic

## Anti-Slop Writing Rules

Do not use:

- repeated "From X to Y" titles
- constant contrast formulas
- em dashes everywhere
- fake-poetic fragments
- 3-word pseudo-profound lines
- inflated claims like "empowers users" or "builds trust seamlessly"
- polished-but-empty PM phrases like "meet users where they are" unless the slide proves it concretely

Prefer:

- full sentences
- takeaway titles that say something real
- concrete nouns and verbs
- product language that a thoughtful PM would actually say aloud
- plain English over fancy phrasing

Bad examples:

- Privacy, Reimagined
- Building Trust at Scale
- From Consent to Confidence
- Empowering Users Through Control

Better examples:

- Asking for less data upfront makes consent more believable
- Backup deserves a different treatment than contact sync
- Optional business uses need a higher bar than core messaging

## Visual Direction

Use an editorial trust-systems look with some product-operator discipline.

That means:

- soft off-white or warm light-gray background
- deep charcoal text
- restrained WhatsApp green accents only where they carry meaning
- high contrast, large type, clean spacing
- very few decorative elements
- no glossy mock enterprise look

The deck should feel:

- considered
- modern
- sober
- premium but not flashy

## Visual Things To Avoid

Avoid:

- bright green everywhere
- giant lock/shield/privacy icons
- endless rounded cards
- fake app-store UI style
- heavy gradients
- dense KPI dashboards on every slide
- overuse of pills, chips, or badges
- too many tiny labels
- icon soup

If a slide works better as clean text with one diagram, do that.

## Typography Feel

Typography should carry the deck.

Guidelines:

- large, confident titles
- body text that is readable without crowding
- clear hierarchy
- not too many text styles

Ideal feeling:

- more editorial report than startup landing page

## Layout Philosophy

Use a few recurring layout families, not a new gimmick on every slide.

Preferred slide types:

### 1. Thesis Slide

- large statement
- one supporting block
- lots of air

### 2. Tension Slide

- left/right or top/bottom contrast
- used sparingly
- focused on one real trade-off

### 3. System Slide

- simple structured framework
- 3 to 4 buckets max
- no messy diagram webs

### 4. Flow Slide

- sequence of decisions or moments
- useful for consent timing and prototype logic

### 5. Metrics Slide

- one main metric area
- a few support metrics
- one guardrail area
- not a fake analytics dashboard

### 6. Rollout/Risk Slide

- compact and practical
- should feel like a PM plan, not a wall of bullets

## Design Principles For This Deck

### 1. Use Restraint

If something is there only to look "designed," remove it.

### 2. Let The Point Carry The Slide

Do not hide weak thinking behind a nice composition.

### 3. Build Visual Contrast Only When It Means Something

Green should not be decorative. It should signal action, choice, or emphasis.

### 4. Keep Complexity Organized

This is a nuanced topic, but the slides should still feel easy to parse.

## The Real PM Point Of View

This is the underlying voice of the deck:

WhatsApp should not treat all data uses the same. Core service processing should be explained clearly, continuity features should be recommended honestly, and optional business-benefit uses should face a higher consent bar.

That point of view should show up repeatedly, in slightly different forms, across the deck.

## Slide-Level Content Rules

Each slide should have:

- one core claim
- one supporting structure
- one reason the audience should care

Each slide should avoid:

- stacking multiple unrelated ideas
- adding bullets just because there is empty space
- repeating content from the prior slide with new wording

## What Will Make The Deck Feel Real

The deck will feel real if it:

- admits trade-offs openly
- treats backup differently from contact sync
- shows that some consent decline can be healthy
- acknowledges business incentives instead of pretending they do not exist
- does not overclaim what a dashboard alone can do

## What Would Make It Feel Fake

The deck will feel fake if it:

- starts with generic privacy language
- uses age-based personas without a good reason
- acts like every user wants maximum control all the time
- ignores WhatsApp's business model
- confuses more prompts with better consent
- uses titles that sound like generated slogans

## Build Standard

Before calling the deck done, it should pass this test:

- Could a smart PM say these sentences in a real interview?
- Could each slide survive a "why does this need to exist?" challenge?
- If the brand colors and logos were removed, would the thinking still feel distinctive?

If the answer is no, revise.
